# Migration: v21 to v22

This guide covers the upgrade from **SDK version 21** (MySQL/MariaDB) to **version 22** (SQLite).

v22 is the first public release with a file-based SQLite database. There is no MySQL-based v22.

## Overview

| Area | v21 | v22 |
| --- | --- | --- |
| Database | MySQL/MariaDB server | SQLite file (`db_path`) |
| Schema setup | MySQL dumps / migrations | `parks_api/database/schema.sql` (automatic) |
| Config keys | `db_hostname`, `db_port`, `db_username`, `db_password`, `db_database` | `db_path`, optional `curl_verify_ssl` |
| Data source | XML export (unchanged) | XML export (unchanged) |

No offer data is migrated from MySQL. The SQLite database is rebuilt from the XML export — same model as before, different storage.

## Before you start

**Prerequisites**

- Running SDK version 21 with a working cron/import
- Valid `api_hash` and `park_id` in `config.php`
- PHP with SQLite support (`pdo_sqlite` / `sqlite3`)
- Write permissions for the web server on `parks_api/data/` (created automatically)

**Backup**

1. Full backup of your `parks_api/` folder (especially `config.php`, `custom/`, custom templates).
2. Optional: MySQL dump for archival purposes only — v22 does not read it.

**Downtime**

Plan a short maintenance window. Pages may show no offers until `migrate.php` completes successfully.

## Step-by-step migration

### 1. Update `config.php`

Keep `api_hash`, `park_id`, and your other custom settings. Remove MySQL keys and add SQLite settings.

**Remove (v21):**

```php
$config['db_hostname'] = '...';
$config['db_port'] = 3306;
$config['db_username'] = '...';
$config['db_password'] = '...';
$config['db_database'] = '...';
```

**Add (v22):**

```php
$config['db_path'] = 'data/park-offers.sqlite';
$config['curl_verify_ssl'] = true; // default; see troubleshooting
```

See [configuration.md](../configuration.md) for all v22 options.

### 2. Replace SDK core files

**CLI alternative** (after step 1 — `config.php` must already contain `db_path`):

```bash
bash parks_api/bin/upgrade-sdk.sh 22
```

This downloads the v22 release, updates core files, and runs `migrate.php`. Skip steps 2–4 below if you use the script.

**Manual:** From the version 22 release ZIP (`swiss-parks-php-sdk-22.zip`), replace:

- `autoload.php`
- `classes/`
- `database/`
- `helpers/`
- `language/`
- `bin/`
- `scripts/` (except if you have local script customizations — rare)

**Keep unchanged:**

- `config.php` (your edited file from step 1)
- `custom/`
- Custom templates under `template/`
- `log/` (if present)

Do not copy `data/` or `log/` from the release package.

### 3. Ensure data directory is writable

```bash
mkdir -p parks_api/data
chmod 775 parks_api/data   # adjust for your host
```

The web server user must be able to create and write `parks_api/data/park-offers.sqlite`.

### 4. Run migration

```bash
php /{PATH-TO-YOUR-API-FOLDER}/scripts/migrate.php
```

This command:

1. Deletes the existing SQLite file (if any)
2. Creates a new database from `database/schema.sql`
3. Runs a full import from the XML export

Use CLI when possible. Browser access to `scripts/cron.php` (legacy alias) remains supported, but migration should run via CLI on production.

### 5. Validate

Check on your site:

- [ ] Filter loads and returns results
- [ ] Offer list displays correctly
- [ ] Map markers appear (if map is used)
- [ ] Offer detail page works
- [ ] Multilingual URLs (if SEO URLs enabled)
- [ ] Favorites (if used)
- [ ] Browser console: no JS errors
- [ ] Logs in `log_directory`: no import errors

Verify cron still works:

```bash
bash parks_api/bin/sync.sh
```

### 6. Cleanup (after successful validation)

- MySQL database and credentials on the server can be decommissioned
- Remove old MySQL config keys from `config.php` if any remain
- Old MySQL dumps in project folders are no longer needed for runtime

### 7. Rollback

If validation fails:

1. Restore the v21 file backup
2. Ensure MySQL credentials are back in `config.php`
3. Confirm v21 cron/import still works against the MySQL database

The v22 SQLite file can be deleted; it is not used by v21.

## Troubleshooting

### HTTPS / cURL errors during import

Some shared hosting environments cannot validate SSL certificates. Symptom: import fails with cURL or SSL errors.

**Fix:** Set in `config.php`:

```php
$config['curl_verify_ssl'] = false;
```

Use only as a fallback. See [configuration.md](../configuration.md).

### Permission denied on database file

Symptom: errors creating or writing `data/park-offers.sqlite`.

**Fix:** Ensure `parks_api/data/` exists and is writable by the PHP/web server user.

### Empty offer list after migration

**Check:**

1. `api_hash` and `park_id` unchanged and correct
2. Migration finished without errors (check logs)
3. Re-run: `php parks_api/scripts/force_update.php`

### «API version out of date» in logs

Run `migrate.php` again after replacing `autoload.php` and `classes/`.

## What stays the same

- XML export as single source of truth
- Integration via `new ParksAPI($language)` in your pages
- `custom/`, templates, favorites, SEO URL configuration
- Sync workflow (`bin/sync.sh` or legacy `scripts/cron.php`)
